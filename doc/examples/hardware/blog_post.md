# Proving a pipeline correct: Ivy, Yosys, and an AI in the loop

*How to describe a synchronous design in Ivy, verify it against an executable
ISA reference, and push the result through Yosys to RTLIL, equivalence checking,
simulation and synthesis — and which parts of that a human actually has to
trust.*

If you already use Yosys for formal verification, you know the shape of the
work: you write a design, you write properties (SVA, or a golden model plus
`equiv_induct`), and you spend most of your time inventing the *helper
invariants* that make an unbounded proof go through. This post is about a
different way to organize that effort, using [Ivy](https://kenmcmil.github.io/ivy/),
and about what changes when an AI assistant writes most of the design *and* most
of the proof for you.

The punchline for the impatient: you describe a pipelined CPU and its
instruction-set architecture (ISA) in Ivy; Ivy proves — by induction, with an
SMT solver, no bound — that the pipeline implements the ISA for *every* program
and *every* initial memory; and `ivy_to_rtl` emits the design as a Yosys RTLIL
netlist you can equivalence-check against SystemVerilog, simulate, and
synthesize. Almost all of the Ivy code in the worked examples was generated by
Claude. The part a human has to read and believe is small, and this post is
partly about drawing that line precisely.

All of the code below is excerpted from complete, checked-in examples; links to
the full files on GitHub are at the end.

## What Ivy is, in two paragraphs

Ivy is a verification language and tool built around *decidable* reasoning. You
describe a system as a state machine, state invariants and other properties, and
Ivy discharges the resulting verification conditions with an SMT solver (Z3).
Its distinguishing design choice is that it steers you toward logic the solver
can decide reliably, so proofs are predictable rather than a black-box gamble —
and when a proof *fails*, Ivy hands you a concrete counterexample-to-induction
(a pre-state/post-state pair) instead of a timeout. It has been used mostly for
distributed protocols, but it has a dedicated sub-language for synchronous
hardware, which is what we use here.

For a Yosys user the mental model is: Ivy is a front end in which you write both
the RTL *and* its specification and proof in one language, get an inductive
correctness proof out, and then translate the RTL half to RTLIL to rejoin your
existing flow. Nothing about the Ivy proof is trusted downstream — the emitted
netlist is ordinary RTLIL, and you are free to re-verify it in Yosys (we do,
below, against an independent SystemVerilog model).

## Hardware in Ivy: registers, wires, clocks

A synchronous design is registers (state updated on a clock edge) joined by
combinational logic (acyclic gate networks that settle between edges). Ivy
models this with exactly three ingredients:

- **`var`** — a state variable, i.e. a register (or, for an array type, a
  memory).
- **`wire`** — a *defined* combinational signal. A wire is a function of state
  and of other wires; its definition may not be cyclic. Crucially, during a
  clock action a wire's value is **frozen** at its pre-edge value — exactly the
  behavior of combinational logic settling before the edge.
- **`export action posedge`** — a clock edge is an exported action (called by
  the environment). Register updates go in an `after posedge` monitor; reset
  goes in `after init`.

Here is a complete 8-bit accumulator — one input `x`, one output `sum`, one
register `acc`:

```
#lang ivy1.8
type word
interpret word -> bv[8]

export action posedge

isolate accum = {
    import wire x : word       # top-level input
    export wire sum : word     # top-level output
    var acc : word
    after init   { acc := 0 }
    after posedge { acc := acc + x }
    definition sum = acc
}
```

Types are bit-vectors (`bv[k]`), `bool`, or arrays (`bv[j] -> bv[k]`, which
become RTLIL memories). Inputs and outputs are not declared as ports; instead a
wire marked `import` is a top-level input and one marked `export` is a top-level
output. Translating this design,

```
$ ivy_to_rtl accum.ivy
```

produces a single RTLIL module in which `acc + x` is an `$add`, `acc` is a
`$dff`, and the `after init` value becomes a reset mux (`rst ? 0 : acc + x`) on
the flop's `D` input. Designs are hierarchical: nested Ivy objects (called
*isolates*) become submodules, and you wire them together with definitions in
the parent. That hierarchy, the memory-initializer story, simulation and
synthesis are all covered in the [Basics](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/basics.md)
document; here we go straight to the interesting problem.

## A pipeline, and the verification problem it poses

Consider a small pipelined CPU. The ISA is seven opcodes (ALU ops, load
immediate, load/store, branch-if-zero) over a 16-bit word, an 8-bit PC, and an
8-entry register file. The two-stage version
([`pipe_cpu.ivy`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/pipe_cpu.ivy))
fetches in stage F and executes in stage X, with a pipeline register between
them:

```
wire fetched : word
definition fetched = imem(pc)          # F: combinational fetch

wire take_branch : bool
definition take_branch = ir_valid & (opcode = 6) & (a_val = 0)   # X: resolved here
```

The branch is resolved in X, one cycle *after* it was fetched — so by the time
we know a branch is taken, the next sequential instruction has already been
fetched into its shadow and must be squashed:

```
after posedge {
    if ir_valid { ... writeback ... };
    if take_branch {
        pc := target; ir_valid := false;     # redirect and squash the shadow
    } else {
        pc := pc + 1; ir := fetched; ir_valid := true;
    }
}
```

This is the classic control hazard, and it is exactly the kind of thing that
makes pipelines hard to verify: at any instant the machine holds several
instructions in different stages of completion, some of which may be about to be
thrown away. What does it even *mean* for this to be "correct"?

The honest answer is: it should behave like the ISA — the simple,
one-instruction-at-a-time reference machine — for every program. Writing a
monolithic inductive invariant that says "this pipelined state corresponds to
*some* ISA state" is possible but painful, and it is precisely the part a Yosys
user recognizes as the time sink. Ivy's hardware examples use a method that
makes this invariant small, local, and — importantly for the AI angle —
mechanically checkable so that a wrong guess is *caught*, not trusted.

## Reference tagging

The method is called **reference tagging**. The idea:

1. Write the ISA as an ordinary executable state machine, `isa_model`.
2. Keep, as **ghost** state, a *trace*: the sequence of architectural states the
   ISA passes through as it executes the program, one entry per instruction.
3. Give each microarchitectural element a **tag** — an index into the trace,
   naming the instruction that element is currently working on.
4. Prove, per element, a one-line invariant: its contents equal the trace value
   *at its tag*.

"Ghost" means proof-only: it exists to state and check the specification and is
stripped out before translation to hardware. (Ivy enforces the separation; more
on that below.)

### The ISA model

The ISA's state is architectural — PC, register file, memory, instruction ROM —
as plain `var`s. The per-instruction intermediate values (the decoded fields,
the operands, the branch decision) are written as **defined functions**:

```
function fetched     = imem(pc)
function opcode      = bfe[13][15](fetched) : opc
function a_val       = rf(ra) : word
function take_branch = (opcode = 6) & (a_val = 0) : bool
```

A subtlety worth calling out, because it is a real gotcha: in the *abstract*
model these are `function`s, not `wire`s. A wire is frozen for the whole clock
action, which is wrong here — within one ISA `step` we want the intermediate
values to track the state as it is mutated. Defined functions re-evaluate on
every state change. The flip side is that to read the value a function had at
the *start* of the action you use `old`, so the PC update reads `old
take_branch` — the branch decision *before* the writeback, so a value written
this step can't retroactively change whether the branch was taken:

```
after step {
    # ... execute / writeback ...
    if old take_branch { pc := target } else { pc := pc + 1 }
}
```

The initial memory contents are *uninterpreted parameters* of the model
(`isa_model(init_mem, init_imem)`). The CPU initializes its memories from the
same functions, so the proof holds for **every** program, not one hard-coded
image.

### The trace, and the tags

The `trace` isolate instantiates the ISA as `arch`, adds an abstract sequence
type `tag`, and records every state the ISA reaches:

```
var now : tag                 # length of the trace so far
var st(T:tag) : state_t       # the recorded states

action step = {
    arch.step;                # advance the ISA by one instruction
    now := now.next;          # extend the trace
    st(now) := current        # record the new architectural state
}
```

Each recorded `state_t` holds the architectural state *and* all the intermediate
values (opcode, operands, branch decision, …) at that point — exactly the
quantities the hardware computes combinationally, so we can pin the pipeline's
internal signals directly to the reference.

Now the payoff. The implementation is the real datapath, augmented with a ghost
tag per stage. Each clock edge we advance the tags to follow instructions down
the pipe, and call `trace.step` exactly when a valid instruction is issued:

```
after posedge {
    x_tag := f_tag;                    # X takes what F held
    if ~take_branch {
        f_tag := f_tag.next;           # F advances only if not squashed
        trace.step                     # ... and a new instruction issues
    }
}
```

And the correctness of each element is a single, local invariant:

```
invariant ~take_branch -> pc = trace.st(f_tag).pc
invariant ir_valid     -> ir = trace.st(x_tag).fetched
invariant rf(R)  = trace.st(x_tag).rf(R)
invariant mem(M) = trace.st(x_tag).mem(M)
```

The fetch PC is the PC of the trace entry at `f_tag`; the execute-stage
instruction register holds the instruction fetched at `x_tag`; the register file
and memory match the architectural state just before that instruction executes.
`ivy_check` proves these hold in every reachable state, by induction. There is
no bound and no hand-written monolithic invariant — the invariant is the
*conjunction of these local facts*, and each one is legible on its own.

## The part you have to trust (and the part you don't)

This is the section a hardware designer working with an AI assistant should read
twice, because it is what makes "the AI wrote the proof" safe.

An Ivy reference-tagging model has four kinds of content, and they differ
sharply in how much human scrutiny they need.

**1. Trusted — the specification.** These you must read and believe, because
they *define what correct means*. If they are wrong, you have faithfully proved
the wrong thing:

- The **ISA model** (`isa_model`): the opcodes, the register-file and memory
  semantics, the branch rule. This is your golden reference. It is small,
  sequential, and looks like a plain interpreter — the kind of thing a human
  architect reads comfortably.
- The **type interpretations** (`word` is `bv[16]`, etc.) and the **clock/reset
  interface** (`export action posedge`, `after init`).
- The **top-level correctness statement**: which implementation state is tied to
  the reference, and under what guard. In the cache example this includes the
  relaxation `~trace.st(trace.now).error -> ...` — "as long as the program has
  not executed stale code, the CPU matches the ISA." You must agree that this is
  the property you want.

**2. Trusted differently — the datapath.** The `after posedge` datapath *is* the
design under verification. You are not asked to believe it is correct — that is
what the proof establishes — but you are relying on the fact that *the thing you
verified is the thing you ship*. Ivy closes that gap: `ivy_to_rtl` emits this
same datapath as RTLIL, and (below) you can independently re-check the netlist
in Yosys. So a bug the AI introduced in the datapath is caught by the proof; a
bug in the *translation* is caught by equivalence checking.

**3. Not trusted — the proof scaffolding.** Everything else is mechanically
checked, which means a mistake in it *cannot* cause a false "verified":

- The **ghost tags and `trace.step` bookkeeping** — which tag advances when,
  when an instruction is issued.
- The **auxiliary invariants** — the per-stage tag invariants above, plus
  structural facts (the pipeline occupancy is a contiguous run of trace indices,
  a mispredicted branch's shadow can only sit in ID, …).
- The **trace's own history invariants** — that consecutive recorded entries are
  related by one ISA step.

If any of these is *too weak*, `ivy_check` fails to prove the property and prints
a counterexample. If any is *false*, `ivy_check` fails to prove *it*. There is no
way for a buggy invariant to make a broken design pass. This is the crucial
asymmetry: the proof is not part of the trusted base. An AI can generate all of
it, get some of it wrong, and the worst outcome is a failed check that you then
iterate on — never a silent false pass.

Ivy makes the ghost/real separation enforceable rather than a matter of
discipline: ghost updates live in `specification { ... }` blocks, the datapath
lives outside, and if ghost state ever leaked into the datapath the translator
would reject it (you cannot build hardware that reads the trace). So "not
trusted" here is backed by the tool, not by a promise.

**4. Boilerplate — auto-generatable.** The trace history invariants in category
3 are the most mechanical: they simply restate the ISA transition relation over
recorded entries. They are voluminous and dull and an LLM produces them directly
from the ISA model. Because they fall in the "not trusted" bucket, generating
them carries no risk beyond a failed check.

So the human budget is: **read the ISA model and the correctness statement.**
That is the spec. Everything else — the datapath (checked by the proof and by
equivalence), the entire proof (checked by `ivy_check`) — can be AI-generated,
and *was*, for these examples.

## The method scales; the reference doesn't change

The same `isa_model` and `trace` isolate are reused **unchanged** across four
designs of increasing complexity. Only the tags and local invariants change.

- **[`pipe_cpu_ref.ivy`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/pipe_cpu_ref.ivy)** — the 2-stage base case above.

- **[`5stage_cpu_ref.ivy`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/5stage_cpu_ref.ivy)** — a standard 5-stage
  (IF/ID/EX/MEM/WB) machine. More instructions in flight, so tags become
  boundary counters; a **data hazard** is handled by stalling EX, which in the
  proof simply means the stalled stage's tag does not advance that cycle.
  Operand correctness then follows from the trace step relation with no extra
  invariant.

- **[`5stage_bp_cpu_ref.ivy`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/5stage_bp_cpu_ref.ivy)** — adds **branch
  speculation**. The machine fetches past unresolved branches, so wrong-path
  instructions enter the pipe and must be squashed, but the trace must record
  only the correct path. A ghost *shadow* bit per stage means "fetched behind a
  mispredicted, not-yet-resolved branch." At fetch, the ghost code compares the
  prediction against the *true* outcome read from the trace and only calls
  `trace.step` when they agree — so the trace never backtracks. The core
  obligation is that shadowed state is never committed, which the local
  invariants make precise.

  The predictor itself is a separate isolate — a nice illustration of Ivy's
  assume-guarantee reasoning:

  ```
  isolate bp = {
      wire fetch_pc : addr          # inputs
      wire br_valid : bool
      wire br_pc : addr
      wire br_taken : bool
      wire predicted_taken : bool   # output
      implementation {
          var bht(I:bhtidx) : ctr   # 2-bit saturating counters (bimodal)
          definition predicted_taken = (bht(pred_idx) = 2) | (bht(pred_idx) = 3)
          after posedge { ... nudge bht toward the resolved outcome ... }
      }
  }
  # in the parent CPU: connect the ports, then read the output
  definition bp.fetch_pc = pc
  definition bp.br_taken = e_take
  definition f_ptaken    = f_is_branch & bp.predicted_taken
  ```

  Because the predictor's logic lives in a hidden `implementation`, Ivy verifies
  the CPU with `bp.predicted_taken` left **arbitrary** — it never looks inside.
  That is sound because CPU correctness does not depend on the *values* the
  predictor produces: a good predictor is faster, but any prediction yields a
  correct execution. The predictor, in turn, has no correctness obligation, so
  its isolate carries no invariants and discharges its (trivial) interface for
  free. The interface the CPU assumes is so weak the predictor meets it by
  construction — assume-guarantee at its most convenient.

- **[`5stage_cache_cpu_ref.ivy`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/5stage_cache_cpu_ref.ivy)** — the one
  design where **the reference model itself changes**. Memory is unified
  (instructions fetched from the same `mem` as loads/stores), and the
  implementation keeps *non-coherent* I- and D-caches (write-back D-cache,
  read-only I-cache) plus a two-cycle memory. A store can leave the D-cache and
  main memory disagreeing with what the fetch path sees — real hardware, and it
  requires software to `FLUSH` before running freshly written code. The ISA is
  extended with a per-address `ddirty` bit and a sticky `error` bit (set on a
  fetch from a dirty address), and *every* implementation-vs-reference invariant
  is guarded by `~error`: once the program executes stale code, the CPU is no
  longer required to match. Each cache is pinned to the reference by local
  invariants in the same style, and — notably — multi-cycle memory costs **no
  new invariants at all**, because "stall ⇒ tag holds" already covers extra fill
  cycles.

The lesson for a verification engineer: the specification (the ISA) is written
once and is stable; the growth in microarchitectural complexity is absorbed
almost entirely by *untrusted, mechanically-checked* local invariants. That is
the ideal division of labor for an AI-assisted flow.

The full method, with all the invariants and the debugging workflow (including
using counterexamples-to-induction to find the missing structural fact), is in
[`reference_tagging.md`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/reference_tagging.md).

## From proof to silicon: back into Yosys

A verified Ivy datapath is only useful if it becomes real hardware, and this is
where the Yosys flow rejoins. `ivy_to_rtl` emits the design as RTLIL, one module
per Ivy object, with array `var`s becoming RTLIL memories, wire definitions
becoming cells, registers becoming `$dff`s, and `after init` becoming
synchronous-reset logic and `$meminit` cells.

Because the output is ordinary RTLIL, **you do not have to trust the Ivy proof
or the translator** — you can re-verify in Yosys. The cache CPU ships with an
independent, hand-written SystemVerilog golden model
([`cpu_golden.sv`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/cpu_golden.sv))
whose register and memory names match the Ivy model, and a Yosys script
([`cpu_equiv.ys`](https://github.com/kenmcmil/ivy/blob/master/doc/examples/hardware/cpu_equiv.ys))
that proves the two combinationally (per-cycle) equivalent:

```
$ ivy_to_rtl 5stage_cache_cpu_ref.ivy
$ yosys cpu_equiv.ys
```

The script reads the emitted `.il` and the SystemVerilog, flattens both
hierarchies (the shared `bp` submodule name makes the predictor's `bht` memory
line up), pairs registers and memories by name with `equiv_make`, expands
memories with `memory_map`, and proves by one-step induction (`equiv_induct`)
that from any equal state the two compute the same next state for every register
and memory. Every matched cell — the scalar registers, all five memories
including the predictor's counter table, and the internal wires — comes back
proven. So the netlist Ivy produced is equivalent, per clock, to an independent
model a second engineer wrote by reading the spec. Two independent paths to the
same design agreeing is a much stronger statement than either one alone.

From there it is a normal Yosys flow. Because `pipe_cpu.ivy` bakes its program
into the memory initializer, the netlist is self-contained and simulates with no
extra stimulus:

```
$ yosys -p "read_rtlil pipe_cpu.il; hierarchy -top cpu; proc; memory_collect;
            sim -clock posedge -reset rst -n 12 -vcd cpu.vcd"
```

and the PC trace after reset is `0, 1, 2, 3, 4, 2, 3, 4, …` — sequential fetch,
then the one-cycle branch bubble (`3, 4, 2`), then the loop. Synthesis is
likewise ordinary (`synth`, `abc -g ...`, or map to a Liberty library). The
Basics document walks through both.

## How this was built, and the skills to do it yourself

Almost every line of Ivy in these four examples — the datapaths, the ghost
scaffolding, and the hundreds of lines of invariants — was generated by Claude,
iterating against `ivy_check`'s counterexamples. The human effort concentrated
where it should: on the trusted surface (the ISA model and the correctness
statement) and on reviewing counterexamples when a proof got stuck. The rest is
either mechanically checked (so AI errors surface as failed checks, not as false
assurance) or independently re-verified in Yosys.

If you want to reproduce this, the Ivy repository ships a **Claude skill**,
[`reference-tagging`](https://github.com/kenmcmil/ivy/blob/master/.claude/skills/reference-tagging/SKILL.md),
that packages the method: the three ingredients, a step-by-step recipe, the
gotchas that each cost real debugging time (defined-functions-vs-wires and
`old`, pinning `bfe` result sorts, shared uninterpreted memory init, packing
cache lines with variadic `concat`, using `shrink=false` and `trace_dir` when
chasing CTIs), and the rules for keeping a model translatable by `ivy_to_rtl`.
It carries the four worked examples as references. Point an agent at it and it
has the context to build and verify a pipeline of this kind.
